Iterative Design

Context

For this project, we were tasked to complete the full iterative design process in solving a design problem for the startup Modular. Specifically, we were given the task to improve the overall onboarding experience for Modular, and through this process, we were able to create initial sketches, develop lo-fi and hi-fi designs through Figma, and receive feedback from our client and our peers in the process. This webpage illustrates the journey that we went through, starting from our initial sketches to our final high-fidelity mockups that we ended up presenting to design leads at Modular.

Defining the Problem

First, the main problem that we had was to improve the Modular onboarding process as Modular found that they were losing around 50% of potential users simply due to the complexity of the onboarding stage. The hypothesis behind this dropoff was due to the fact that the onboarding process was complicated and long, involving the use of a terminal to run commands. Additionally, the old onboarding process brought the user to a plain documentation page. Therefore, there were two main problems that we wanted to solve:


  1. Avoid having the user simply just visit the Modular documentation page.
  2. Figure out if we want the user flow to be "Create an Account" and then "Install the SDK" or "Install the SDK" first and then "Create an Account."

Therefore, with these two main pain points in mind, we started our initial design process.

Lo-Fi Wireframes

After compiling all of our sketches, we incorporated elements from all of our sketches, and we started to create our first iteration of a low-fidelity wireframe. Click the button below and navigate to the "Low-Fidelities" page to see the first iteration of our Lo-Fi wireframe.

Design Decisions: Here are some of the major design decisions that were made:

  • Flow One vs. Flow Two: Initially, we made two user-flows. One was focused on having the user create an account and then install the application locally. We called this Flow One. The second user-flow had the user install the application locally first, test it, and then create an account afterwards. We called this Flow Two. We felt that the former would be users that have experienced the Modular application before while the latter would be helpful for users that haven't played around with the application before. Therefore, this is what led us to our main proposed design.
  • Forked User Flow: Our main design that we reflected in our Lo-Fi Wireframe was actually to combine Flow One and Flow Two to create a Forked User Flow that appeals to users that are both familiar and unfamiliar with the platform. Instead of choosing which user-flow that would appeal to all users, we made the design choice to give that choice to the user so that they can get the most out of the platform based off their familiarity with the product. Then at the end, all users will end at the same position which would be the user's personal dashboard.
  • User Dashboard: The final main design choice was changing the user's personal dashboard after they created an account to be a checklist to ensure that the user has followed all of the necessary steps to set up the application locally. This also solves the problem of just directing the user to the documentation page.

Part 2: Critique

One of the most important parts of this project was receiving feedback on our design. First, we were able to film a five minute Loom video that demonstrates our design flow.

Next, we presented our Loom video to our peers in our Critique Design Studio. After presenting, our peers from different groups provided feedback on our initial design flow. Additionally, we presented our Loom video to our client who provided us with additional feedback. Finally, we also made a presentation to Vanessa who provided some comments as well. We've collected our feedback points from our client, our peers, and Vanessa in the following feedback list below:


Feedback:

  • [Peer]: "I like the checklist interface for onboarding. I'ts better than the documentation, and I would prefer this as someone not familiar with Modular.""
  • [Peer]: "The documentation is a super long work flow, so people dont want to get through the entire process and never end up making the account - having both options could be good."
  • [Peer]: "The phrase, "learn more" is slightly confusing, and the fact that the docs page is a different page could be a reason why users are dropping off."
  • [Peer]: "Might be useful to give preview of what learn more is going to do, rather than just learn more, give more info about what to expect."
  • [Peer]: "Maybe it could be possible to automate the process for the user (let the program run the commands for the user)"
  • [Vanessa]: "Maybe there could be an option of doing both flows, so one flow could be for people that know about Modular to immediately just create an account, and one flow could be for people that don't know too much about Modular, so they go through the "learn more" route."
  • [Client]: "If you want to psychologically nudge users towards account creation, then don't give the 2 options the same visual weight. Make the 'create account' feel like the primary choice, and add some ROI to it 'Recommended option' or 'This option allows you to use Free credits for production deployment.'"
  • [Client]: "I'd rebrand "Learn More" as 'test MAX locally first' or some language that is a stronger outcome than just learning, but implies that it's another way to try MAX."

Part 3: Wireframe II

Through the feedback that we've received in the section above, we made some changes to our Lo-Fi wireframes that are found below:

Update Decisions: Here are some of the major updates that were made:


  • First, per our client's request, we changed the "Learn More" button at the beginning of the flow to "Test MAX Locally First."
  • Second, on the "Create an Account" page, we added information as to why the user should be incentivized to create an account (i.e., free credits to deploy their model to the cloud).
  • Third, we changed the visual hierarchy of the "Create an Account" button when compared to the "Test MAX Locally First" to try to incentivize users to create an account early.

Overall, these were three of the main changes that we made in this second iteration.
